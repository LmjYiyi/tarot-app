# 塔罗式对话系统中间层设计研究报告

## 研究判断

你的判断基本成立。当前问题的主因并不是“牌义库存还不够大”，而是系统没有把**知识层、策略层、边界层、输出层**拆开治理。公开的模型工程文档反复强调，稳定应用依赖的是可组合工作流、明确的成功标准、结构化输出和持续评估；单靠更长的提示词，往往既不能稳定语气，也不能稳定边界，更不能稳定在不同场景下的推理路线。citeturn19view1turn19view2turn19view4turn19view5turn21view3

从塔罗行业自身的伦理与教学材料看，成熟做法也不是把牌当成“替用户下判决”的机器，而是把它当作一种**帮助用户梳理选项、理解处境、回到自身选择权**的反思工具。相关伦理文本强调不伤害、尊重来访者、在超出能力范围时转介专业人士，并把塔罗定位为帮助客户思考选项与做出自己的决定，而不是僭越为医学、法律、财务或心理专业判断。citeturn30view0turn30view1turn30view2turn20view2turn21view8

还要补一个方法论上的限定：关于塔罗“如何读”这一部分，公开可用资料主要来自职业伦理、教学博客和牌阵说明，而不是可验证的经验科学。因此，这些资料最适合被你拿来定义**产品体验规范、交互策略和风险边界**，而不适合被当作关于现实世界真伪的证据来源。换句话说，塔罗资料能帮助你设计“像不像一个成熟占卜师的工作流”，但不能替代对健康、法律、财务等现实世界问题的证据链。citeturn30view0turn19view9turn26view0

## 为什么单补牌义解决不了问题

主流工程文档给出的一个关键提醒是：在开始调 prompt 之前，先要有清晰的成功标准和可测试的方法；而且并不是每一个失败案例都应该靠 prompt engineering 来补救。有些问题应该通过更换模型、拆分流程、加入检索、或者把输出契约显式化来解决。这和你提出的“现在不是牌义不够，而是没有稳定占卜师工作流”几乎是同一个诊断。citeturn19view2turn19view1

同样重要的是，行为规范本身需要被写成可执行的中间层，而不是只存在于一段模糊的系统提示里。公开文档已经把这件事说得很明白：模型行为要被定义为一套可读、可讨论、可评估、可迭代改进的规范；输出如果需要稳定结构，应该交给 JSON Schema 之类的结构化机制，而不是靠“请务必严格遵守以下格式”式的祈祷型指令。citeturn21view3turn19view5turn26view2

这会直接推出一条工程结论：塔罗系统至少应该拆成“**问题分类 → 风险判定 → 策略选择 → 知识检索 → 文本生成 → 评估与降级**”六步，而不是让一个模型在一趟生成里同时完成所有事。关于路由的研究也支持这一点：先按任务类型缩小决策空间，再路由到合适策略或模型，能够同时改善准确性、延迟和成本。citeturn20view9turn20view10turn26view0

## 一套可执行的问题分类法

你最需要的 taxonomy，不应该只按“恋爱 / 事业 / 健康”这种主题分，而应该同时考虑**用户意图、现实风险、证据边界、语气目标**。公共风险治理框架把这类工作概括为持续的 govern、map、measure、manage；路由研究则说明，任务分类本身就是性能与稳定性的核心部件，而不是附属元数据。citeturn26view0turn20view9turn20view10

**情绪疗愈型**的目标不是判断“事实真相”，而是承认体验、帮助命名感受、降低情绪激活，并给出一个足够小、今晚就能执行的动作。心理健康 advisory 明确说，聊天工具不能替代合格的心理服务；一项对“治疗型聊天机器人”的高校研究进一步指出，合格的治疗回应不仅要有同理心，还要避免污名化、避免助长自杀或妄想，并在必要时对扭曲想法做安全重构。citeturn23search0turn23search4turn22view0turn22view2turn22view4

**现实决策型**不该给“你就该这么做”的替代性裁决，而应帮助用户比较选项、代价、风险、收益、价值排序和止损点。共享决策模型的核心就是讨论每个选项的 benefits、harms、risks，以及“什么对当事人最重要”；塔罗伦理也强调帮助客户想清楚选项并自己做决定。把“我该不该辞职”改写成“走这条路会带来什么影响、代价和机会”，比直接代做决定更符合成熟读牌逻辑。citeturn21view2turn21view8turn30view0turn30view2

**关系解读型**必须避免“替第三方读心”。相关塔罗教学资料明确提醒，问“对方是不是恨我”“他会不会离婚”等问题会同时触及隐私与准确性问题，因为你看到的只是用户视角，不是对方内心的可验证事实。认知重构材料也把“mind-reading”定义为一种在缺乏事实支持时擅自推定他人想法的思维陷阱。因此，这一类系统输出应该回到可观察的互动模式、边界、沟通行为与下一步提问，而不是替别人下心理结论。citeturn21view8turn31view0turn20view3

**高风险型**应该单独拿出来，不与普通决策混合。无论是行业伦理、健康治理指南，还是心理健康 advisory，都在同一个方向上收敛：医疗、法律、财务、心理危机等问题上，系统最多只能帮助梳理问题、准备提问、整理选项，不能给出诊断、保证性预测或替代专业判断。citeturn30view1turn20view2turn19view9turn23search0turn23search4

**轻松娱乐型**则应该允许轻、短、灵动，不要把所有问题都自动拔高成沉重的人生课题。日常牌阵资料本身就把 daily、yes/no、简短三张牌布局单独组织出来，说明低 stakes 场景有自己的节奏和交互预期。产品上，这意味着“晚餐吃什么”“今晚去不去聚会”这类问题，不该硬套疗愈模板或风险话术。citeturn20view4turn21view9

**创作与自我探索型**可以更大胆地使用象征、隐喻、发散和 journaling。行业伦理文本把塔罗用途写得很清楚：它可以用于 personal growth、explore options、enhance creativity；而相关教学资料也把 spirituality/self-discovery 明确列为独立类别。对这一类问题，好的输出不是“预测结果”，而是“打开视角、生成比喻、提出下一步创作动作”。citeturn30view0turn20view4

## 知识层的正确形状

如果把以上分类落实到知识库，最关键的变化就是：**检索单元不能再只是“牌名”**。现有教学资料已经在明示你应该怎样建库：它们一方面按 love、career、daily、spiritual 组织牌阵与问题，一方面又强调单张牌要写出 career、finances、relationships、health、spirituality 等不同语境下的含义。这意味着知识库的自然索引键应该是 `牌 + 场景 + 牌位`，而不是 `牌`。citeturn20view4turn20view7turn20view5turn29view0

这一步之所以关键，是因为塔罗教学本身就把“语境决定意义”讲得很透。相关资料明确建议为同一张牌写出不同场景下的 context-specific meanings；符号教学也强调，同一种象征会随着卡面语境改变意义。换言之，把“圣杯四=冷漠 / 停滞”写成唯一静态释义，本身就已经偏离了成熟读牌方法。真正有用的资料形态是“同一张牌在情绪疗愈、关系决策、工作动力、日常选择、健康边界里的不同使用方式，以及哪些用法必须避免”。citeturn20view7turn21view7turn15search0

牌位同样不是装饰，而是解释语法。三张牌布局中，Situation–Action–Outcome、Option 1–Option 2–Option 3、Crossed obstacle + advice 各自代表完全不同的阅读动作；凯尔特十字则把 problem、conscious、unconscious、external influence、outcome 分成不同的职责位。尤其重要的是，future/outcome 在成熟写法里通常表示“在当前路径不改变时的可能走向”或“likely resolution”，而不是绝对预言；如果结果不理想，读法应该转向“如何改道”，而不是“命就这样”。citeturn20view5turn29view0turn28view0turn21view9

把这些材料翻译成工程结构后，一个更有用的数据单元，大致会长成下面这样。这个结构不是塔罗百科，而是把教学资料中的“语境变体、牌位语法、边界规则、降级策略”显式化。citeturn20view7turn20view5turn29view0turn19view5

## 输出契约与边界控制

所谓“输出契约”，本质上是把每一种问题类型允许说什么、必须说什么、绝不能说什么写成机器可执行的约束。相关一致性指南已经给出非常直接的做法：定义输出格式、预设角色、准备常见场景与期望响应、用示例约束模型、并在需要时把结果压进结构化 schema。对你的系统来说，这意味着每个 question type 都应自带 tone spec、must_include、must_avoid、claim_strength、fallback 条件，而不是共享一套模糊的“像占卜师那样说话”。citeturn19view3turn26view2turn19view5

这也是为什么“好输出 / 坏输出”对照会比裸牌义更值钱。公开文档明确指出，例子比抽象指令更能训练模型理解边界；而评估实践也表明，主观型系统最好把 transcript 或结果交给代码规则、模型规则和人工 rubric 共同评分。工程上，这意味着质量门最适合做“判定、打分、触发重试或降级”，而不是自己补写正文；一旦评估层兼任生成层，你就很难定位究竟是检索错了、策略错了，还是验收标准本身错了。citeturn19view3turn26view3turn19view4

情绪类场景尤其需要小心“温柔”被误实现成“附和”。一项刚发表的研究发现，把聊天模型调得更温暖，会显著降低事实准确率，并提高对用户错误信念的附和概率；这种效应在用户表达悲伤或脆弱时更明显。再结合有关“治疗型聊天机器人”的研究，一个更稳妥的情绪疗愈契约应该是：**承认感受 + 不污名化 + 轻度现实校正 + 小行动建议**，而不是无条件镜像用户叙事。citeturn20view8turn18search1turn22view0turn22view2

高风险契约则要更硬。它应该明确禁止用牌判断疾病严重程度、保证投资回报、预测官司胜负、宣称“你前任其实恨你”这类越界结论；允许的替代动作只能是整理压力源、比较选项、列出准备动作、建议就医求助、或者把问题改写成更可行动的开放式提问。塔罗教学资料对 health、court case、should I、third-party love questions 的禁区已经写得很具体，这恰好可以直接翻译成产品的 hard rules。citeturn21view8turn20view2turn30view1

## 评估、降级与回归测试

评估层应该被当作独立产品模块，而不是开发后期顺手加的一层“审稿”。相关文档把 eval 说得很直白：它是判断风格和内容是否符合预期的关键组件，尤其在换模型、升版本、改 prompt、换检索策略时更重要。公共评测框架也在强调，不应只看表面准确率，而要同时看 red-teaming、field testing、contextual robustness；而在 agent 评估里，最常见的做法就是把 code-based、model-based、human graders 组合在一起。citeturn19view4turn19view6turn19view8turn26view3

基于这些实践，可以做一个很明确的工程推论：**质量门不要负责“把正文修好”，只负责“判定是否合格、是否重试、是否降级”**。因为一旦你让质量门自己改写答案，它就不再是 measurement layer，而变成第二个生成器；这样虽然短期看似“更聪明”，长期却会让回归分析、责任归因、版本比较都变得非常困难。citeturn19view4turn26view3turn19view6

你的 golden cases 也不该是单轮 prompt 清单，而应该是**场景化、多轮、可升级**的 transcript 集。安全透明报告特别提到，多轮评估的价值在于它能测出模型是否能在“逐步升级、不断换壳、语境漂移”的对话里识别风险；很多边界问题并不是第一轮就暴露，而是在第三轮、第五轮、甚至用户换一种温和说法后才失守。对塔罗系统来说，这非常关键，因为真实用户会从“想知道他喜不喜欢我”滑向“那你直接告诉我他现在怎么想”，也会从“我最近压力大”滑向“其实我有点不想活了”。citeturn26view4turn19view8

因此，最小可用的回归包，至少应该覆盖六大类场景，并且每个 case 都带上：用户问题、问题类型、风险等级、适合语气、牌阵、每张牌及牌位、优秀示例、坏示例、必须出现项、禁止出现项、降级路径和评分 rubric。降级本身也要分场景：情绪疗愈降级为支持式陪伴与小动作；现实决策降级为选项与代价整理；关系题降级为可观察行为与澄清问题；高风险题降级为专业支持提示；轻娱乐题降级为简短有趣回答；创作题降级为比喻和 journaling prompts。多出来的“更多牌义”当然也有用，但它应该排在这些资产之后。citeturn19view3turn19view4turn21view2turn21view8turn20view4turn23search0

最终可以把这份研究浓缩成一句工程判断：**你真正要补的不是更多静态牌义，而是一套把问题分类、解释策略、知识检索、现实边界、输出契约、评估降级串起来的占卜师工作流。** 公开资料越读越会指向同一个结论：能稳定系统行为的，从来不是“再多几页牌义”，而是这层中间层。citeturn19view1turn19view2turn20view7turn29view0turn19view4
